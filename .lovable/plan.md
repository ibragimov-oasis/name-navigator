## Что добавляем

Сейчас на сайте есть `/children` (детские имена) и `/pets` (для питомцев), но нет полноценного раздела **«Для людей»** — для взрослых, для себя, для подбора имени по личности, для смены имени, для творческих псевдонимов, для именования персонажей и для имени-в-исламе (мусульманское имя для новообращённого). Это критическая дыра — большая часть аудитории сайта (особенно мусульманской) ищет имя именно для **взрослого человека или для себя**, а не только для младенца.

Также строим инфраструктуру **профиля человека** — единая сущность «Person», вокруг которой работают тафсир, нумерология, ДНК имени, подпись, насаб и совместимость. Это превращает сайт из набора отдельных инструментов в единый AI-помощник по работе с именем человека.

## План — 5 фаз

### Фаза 1. Раздел «Для людей» (`/people`)

Новая главная страница раздела с подразделами-карточками:

1. **Подбор имени для взрослого** — тот же каталог, что `/children`, но с фильтрами «возраст», «цель» (смена имени / новообращённый в исламе / творческий псевдоним / для персонажа), «впечатление имени» (солидное, мягкое, нейтральное, харизматичное).
2. **Мусульманское имя для новообращённого (شهادة)** — отдельный воркфлоу: пол → характер → желаемое значение (благодарность, свет, вера, путь) → 5 рекомендаций с тафсиром, дуа при принятии имени, инструкция-шаги.
3. **Имя для персонажа** — для писателей, RPG, бренд-неймов: культура → эпоха → архетип (герой, мудрец, тень, бунтарь) → имя + история + фамилия + псевдо-биография через AI.
4. **Псевдоним / стейдж-нейм** — короткое, звучное, доменно-доступное (проверка `.com` опционально позже).
5. **Историческая личность** — поиск среди реальных людей (пророки, сахабы, ученые, поэты, правители) с биографией и причиной известности.

Файлы: `src/pages/People.tsx`, `src/pages/people/AdultNames.tsx`, `src/pages/people/RevertName.tsx`, `src/pages/people/CharacterName.tsx`, `src/pages/people/Pseudonym.tsx`, `src/pages/people/HistoricalFigures.tsx`.

В `Header.tsx` добавить пункт «Люди» (icon: Users) между «Дети» и «Питомцы». На `Index.tsx` — добавить карточку в основную сетку фич.

### Фаза 2. Сущность «Person» и хранилище профилей

Единая модель `Person` для всех инструментов:

```ts
type Person = {
  id: string;
  fullName: string;          // Имя
  surname?: string;
  patronymic?: string;       // отчество / насаб
  kunya?: string;            // Абу/Умм
  nisba?: string;
  laqab?: string;            // прозвище
  gender: 'male' | 'female';
  birthDate?: string;        // для нумерологии судьбы
  relation: 'self' | 'child' | 'spouse' | 'parent' | 'friend' | 'character' | 'historical';
  notes?: string;
  createdAt: number;
};
```

Хранение: **локально в localStorage** (как `favorites`) + опционально синк в Supabase, когда инфраструктура восстановится. Provider `PeopleProvider` по образцу `FavoritesProvider`.

Страница `/people/profiles` — список «моих людей» с быстрыми действиями: тафсир, нумерология, ДНК, подпись, совместимость.

Файлы: `src/lib/people.tsx` (provider + storage), `src/pages/people/Profiles.tsx`, `src/components/PersonCard.tsx`, `src/components/PersonForm.tsx`.

### Фаза 3. Совместимость двух человек (`/people/compatibility`)

Совместимость по имени **между двумя людьми** (супруги, родитель-ребёнок, бизнес-партнёры):
- фонетическая гармония (уже есть `nameHarmony.ts` — используем);
- нумерологическая совместимость (Абджад: разница чисел судьбы → балл);
- культурно-религиозная совместимость (одна культура / совместимые / разные);
- атрибутный матч (общие черты vs дополняющие);
- AI-объяснение через `rag-query` («почему пара Мухаммад + Аиша работает: …»).

Результат — карточка с общим баллом, радар-чартом и 3 пунктами «сильные стороны / зоны внимания / совет».

Расширяем существующий `/compare` либо делаем отдельную страницу с акцентом на «два человека», а не «два имени-варианта».

### Фаза 4. AI-поиск и RAG для людей

Расширяем `knowledgeIndex.ts` новыми типами документов:
- `revert-guide` — гайды по принятию ислама и выбору имени;
- `historical-figure` — биографии известных мусульман и не только;
- `name-impression` — психология восприятия имени (солидное / мягкое / харизматичное).

В `AISearchDialog`:
- новые быстрые фильтры: «для себя», «для персонажа», «для новообращённого»;
- AI отвечает с учётом контекста выбранного `Person`, если есть активный профиль («подскажи имя для моего сына, родился вчера, мусульманин» → подтягивает профиль и фильтрует).

Edge function `rag-query` уже принимает контекст — добавить поле `personContext` в payload.

### Фаза 5. Дополнительное, чего не хватает сайту

Параллельно с «людьми» закрыть очевидные дыры:

1. **SEO и мета** — у каждой страницы свой `<title>` и `<meta description>` (сейчас всё дефолтное), Open Graph для шаринга карточки имени, sitemap.xml, robots.txt уже есть.
2. **Шаринг имени** — кнопка «Поделиться» на `NameCard`: красивый OG-preview (имя + значение + культура), копирование ссылки `/?name=Мухаммад`.
3. **Pронунсиация (TTS)** — кнопка озвучки имени через Web Speech API + arabic transliteration уже есть.
4. **Авторизация (опционально, после восстановления Supabase)** — чтобы профили `Person` и `Favorites` синкались между устройствами.
5. **Экспорт PDF** — «Сертификат имени» с тафсиром, насабом и дуа, для сохранения / печати.

Каждый пункт — небольшая задача, закрывает реальную пользовательскую боль.

## Технические детали (для разработчика)

- Все новые страницы — React + Vite, использовать `Header`, дизайн-токены (`coral-light`, `teal-light`, `lavender-light`, `gold`).
- Никаких новых зависимостей — переиспользуем `lucide-react`, `react-router-dom`, `@tanstack/react-query`, существующие UI-компоненты `shadcn`.
- `PeopleProvider` — `localStorage` ключ `imyagen.people.v1`, миграция при изменении схемы.
- `RevertName` workflow — мульти-шаговый wizard, переиспользуем паттерн из `NameWizard.tsx`.
- `CharacterName` — генерация через `rag-query` с режимом `creative=true` (новый параметр в edge function).
- `HistoricalFigures` — берём данные из `prophets` + extend новым `src/data/historicalFigures.ts` (учёные, поэты, халифы).
- Совместимость — чистая клиентская логика в `src/lib/personCompatibility.ts`, без бэкенда.
- Все профили `Person` валидируются через лёгкую схему (можно `zod`, уже в проекте через shadcn forms).

## Что НЕ делаем сейчас

- Не трогаем Supabase pgvector (ждём восстановления инфраструктуры — это в Фазе 6 предыдущего плана).
- Не делаем платную подписку / монетизацию.
- Не делаем мобильное приложение / PWA-installer.
- Не переписываем существующие `/children`, `/pets`, `/wizard` — только аккуратно дополняем.
